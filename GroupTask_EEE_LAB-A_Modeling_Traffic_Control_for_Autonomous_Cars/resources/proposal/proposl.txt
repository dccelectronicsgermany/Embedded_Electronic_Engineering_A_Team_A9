\section*{Purpose of this Document}
This document proposes a shared system concept for our autonomous intersection project. The goal is to align the whole team before we update the requirements diagram, use-case diagram, activity diagram, sequence diagram, UPPAAL timed automata model, and later the FreeRTOS / HW--SW implementation. The main problem we are solving is not simply ``which car goes first?'' but rather how autonomous vehicles can safely and efficiently share an intersection that contains multiple conflict areas.

\section{Core System Idea}
We model the autonomous intersection as a real-time shared-resource system. The intersection is the shared resource, vehicles are concurrent processes, and the conflict areas inside the intersection are critical zones. A vehicle may cross the intersection only according to a reservation schedule assigned by the controller.

\begin{quote}
\textbf{Main safety rule:} No two vehicles may occupy or reserve the same critical zone during overlapping time intervals.
\end{quote}

This means the system is not limited to allowing only one vehicle to cross. Multiple vehicles may cross simultaneously if their required critical-zone sequences do not conflict in time.

\section{Vehicle Arrival and Speed Control Assumption}
To keep the model understandable, we assume each head-of-line vehicle enters the control zone at a fixed distance with a known starting speed. This makes arrival-time prediction simple enough for a timed automata model.

Let:
\begin{itemize}
    \item $D$ be the fixed distance from the detection line to the first critical-zone entry line,
    \item $v_0$ be the starting speed when detected,
    \item $v_{max} = 2v_0$ be the maximum allowed speed,
    \item $v_{min}$ be the minimum smooth speed before the vehicle must stop.
\end{itemize}

The controller can compute the reachable arrival window:
\[
T_{earliest} = \frac{D}{v_{max}}, \qquad
T_{normal} = \frac{D}{v_0}, \qquad
T_{latest} = \frac{D}{v_{min}}.
\]

The controller searches for the earliest non-conflicting critical-zone schedule between $T_{earliest}$ and $T_{latest}$. If no schedule exists within this window, the vehicle should stop before entering the intersection.

\section{Time-Slot Reservation Concept}
Each vehicle receives a schedule for its required zone sequence. A vehicle does not need all zones at the same instant. It only needs each zone during the specific interval in which it passes through that zone.

\section{Scheduling and Priority Policy}
Only the first vehicle in each incoming lane is active. Since there are four incoming lanes, the controller compares at most four active vehicles at a time. Vehicles behind the first vehicle in a lane cannot request access before the lane leader has moved.

The controller should use a simple and explainable priority policy:
\begin{enumerate}
    \item Emergency vehicles have highest priority.
    \item Normal vehicles are prioritized by smallest slack time.
    \item If slack is similar, the vehicle with longer waiting time has priority.
    \item If still equal, use FIFO order.
    \item Non-conflicting vehicles may be scheduled together to improve throughput.
\end{enumerate}

Slack time can be understood as:
\[
\text{slack} = T_{latest} - T_{assigned\_entry}.
\]
A small slack means the vehicle is close to the point where it can no longer cross smoothly without stopping. However, priority never overrides safety. A vehicle may only receive a slot if its required critical zones are free during the assigned intervals.

\section{Communication Roles: V2I and V2V}
V2I is the main scheduling communication path. Vehicles send requests to the controller, and the controller sends schedules back to the vehicles. V2V is not the main scheduler; it supports safety awareness.

\begin{center}
\begin{tabular}{p{0.22\linewidth}p{0.68\linewidth}}
\toprule
\textbf{Communication} & \textbf{Role in our model} \\
\midrule
V2I & Vehicle sends route intent, speed, position and lane to the controller. Controller returns zone schedule and speed command. \\
V2V & Vehicle broadcasts intent, emergency state or warning information to nearby vehicles for awareness and fail-safe behavior. \\
\bottomrule
\end{tabular}
\end{center}

\section{Suggested UML Updates}
\subsection{Use-Case Diagram}
The updated use-case diagram should include the following use cases:
\begin{itemize}
    \item Detect approaching vehicle
    \item Declare vehicle intent
    \item Calculate arrival window
    \item Map route to critical zones
    \item Reserve critical-zone time slots
    \item Send speed command
    \item Enter critical zone according to schedule
    \item Stop before intersection if no feasible slot exists
    \item Broadcast intent or emergency state through V2V
\end{itemize}

\subsection{Activity Diagram}
The activity diagram should follow the controller algorithm shown in Figure 4. The main decision point should be: \textit{Is a non-conflicting and reachable schedule available?}

\subsection{Sequence Diagram}
The updated sequence diagram should include these lifelines:
\begin{itemize}
    \item Vehicle
    \item Vehicle local controller
    \item Intersection controller
    \item Reservation table
    \item Critical zones
    \item Other vehicles for V2V awareness
\end{itemize}

Important messages:
\begin{itemize}
    \item Vehicle $\rightarrow$ Controller: request(route, speed, distance, lane)
    \item Controller $\rightarrow$ Reservation table: checkAvailable(routeZones, arrivalWindow)
    \item Reservation table $\rightarrow$ Controller: earliest feasible schedule
    \item Controller $\rightarrow$ Vehicle: schedule(zoneTimes, speedCommand)
    \item Vehicle $\rightarrow$ Critical zone: enter only if reservation is valid
    \item Vehicle $\rightarrow$ Controller: zone exited / crossing complete
\end{itemize}

\section{UPPAAL Modelling Plan}
UPPAAL should model a network of timed automata. The first improved version should stay abstract and verifiable instead of modelling full vehicle physics.

\subsection{Templates}
\begin{itemize}
    \item \textbf{Vehicle template:} Idle, Detected, Requesting, ControlledApproach, WaitingBeforeEntry, InCriticalZone, Exited.
    \item \textbf{Controller template:} ReceiveRequest, CalculateWindow, MapRoute, FindSlot, ReserveSlots, SendCommand, EmergencyMode.
    \item \textbf{Reservation table logic:} Abstract arrays storing zone ownership per time slot.
    \item \textbf{Optional emergency template:} Creates emergency request and override condition.
\end{itemize}

\subsection{Discrete Speed Modes}
Instead of continuous acceleration, use discrete speed modes:
\begin{center}
\begin{tabular}{ll}
\toprule
\textbf{Mode} & \textbf{Meaning} \\
\midrule
FAST & Reach first critical zone at earliest arrival time. \\
NORMAL & Reach first critical zone at normal arrival time. \\
SLOW & Reach first critical zone near latest smooth arrival time. \\
STOP & Stop before the first critical zone if no feasible slot exists. \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Properties to Verify}
\begin{itemize}
    \item No two vehicles occupy the same critical zone at the same time.
    \item A vehicle enters a critical zone only if it has a valid reservation.
    \item The system is deadlock-free.
    \item Emergency priority does not create unsafe conflicts.
    \item A vehicle that cannot be scheduled smoothly stops before the intersection.
\end{itemize}

Example UPPAAL query ideas:
\begin{verbatim}
A[] not deadlock
A[] no_two_vehicles_same_zone
A[] vehicle_in_zone imply reservation_valid
A[] emergency_active imply no_conflicting_normal_vehicle_enters
\end{verbatim}

\section*{Proposed Team Decision}
The team should agree that our improved model is a critical-zone time-slot reservation system with speed-controlled arrival. This gives us a clear structure for requirements, UML diagrams, UPPAAL verification and implementation mapping. It also gives us a strong explanation of why multiple vehicles can cross at the same time while still guaranteeing safety.
